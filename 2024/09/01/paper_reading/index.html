240901

Leap

ATC’20 best paper

关键词：dynamic memory prefetch；memory disaggregation；RDMA；algorithm

一句话概括：提出一个可以预测stride pattern并且容忍部分irregular pattern的memory prefetch算法，将其应用在RDMA的data path之上，取得了很好的效果。

  1. Main Content

    1. Background

      Memory Disaggregation 是一种数据中心架构技术，它将计算资源和内存资源分离，允许它们独立扩展和管理。传统的数据中心服务器通常将计算、存储和内存资源紧密耦合在一起，每个服务器都有自己的处理器、内存和存储。Memory Disaggregation 打破了这种耦合，通过将内存资源独立出来，使得多个计算节点可以共享一个更大、更高效的内存池。利用RDMA来扩大本地内存，这样一来就可以用RDMA的网络请求来代替磁盘交换了。

    2. Observation

      通常借助VFS或者VMM接口，data path效率低下，breakdown：

    3. Solution

      创建一个新的data path -> 实现读、写、预取等功能 -> 如何高效预取？

      预取算法：trend detection（stride sequence） + 慢启动/拥塞控制

      1. 规定初始化window size

      2. 选取window中的major element（重复次数大于window size / 2 + 1）；若无则扩大window（x2）继续找，直到找到或者到达最大窗口大小。

      3. 预取

        上次cache命中情况？

        1. 命中少/不命中

          和缓缩减预取页数大小（/2）

        2. 命中多

          扩大预取页数（x2）

  2. Comments

    1. 分析：

      1. 亮点：算法表现好，可以防止irregular的访问，+对stride pattern的支持
      2. 亮点：算法设计思想，对stride pattern的理解，对“慢启动+拥塞控制”的应用
      3. 疑似包装：我其实不大懂这预取算法有用到RDMA的什么特性吗？看下来感觉这两点有点点割裂，不过其实也还好：
        1. 独立data path加速
        2. 提出新的预取算法

    2. 启发：

      1. 问题很有意思

        Memory Disaggregation感觉非常完美地代表了传统system的美学，“分布式的固定分配->集中分配”，恰如传统OS的“分段->内部碎片严重->段页结合”；“集中分配->互相直接窃取(RDMA)”，也正如“运算器为中心->DMA”。

        然后，本文又聚焦于类似传统内存与磁盘之间的预取问题，提出新的预取算法并且应用到新场景下，着实眼前一亮。

      2. 思想值得学习

        1. 独立data path：当你发现某个过程需要经历的层数多得可怕，不如想想可否对它进行一个“垂直整合”
        2. 问题建模：内存页排队换入换出<->网络请求排队

DMon

OSDI’21

关键词：memory profiling；static cache prefetch；data locality analysis

一句话概括：通过memory profiling，找到程序中具有数据局部性问题（包括cache、内存换入换出）的代码段，自动对其进行代码修复，重新编译，开销低。

  1. Main Content

    1. Background

      memory profiling通用流程：

      1. 收集历史信息
      2. 根据历史信息提取出data pattern等信息，设计解决方案
      3. 执行解决方案

      现有技术：

      1. compiler
      2. dynamic profiling
      3. static sampling

      问题：

      1. 需要人工修改，耗费时间
      2. 静态分析不足以支撑动态执行
      3. 动态分析+修改开销太大

    2. Observation

      Intel top-down method：决策树进行开销拆分、逐层排查问题。

    3. Solution

      1. 收集历史信息

        目标：确认是哪一条访存指令的锅+根据该指令确认程序出错位点

        1. 监控决策树子集

        2. 分层分时采样

          跟DAMON差不多，隔一段interval（time slice）进行一次性采样，如该段时间内信息量满足决策树需求，再进入下一层采样

        3. 获取优化位点

          在Layer 4获取高频访存指令对应的源码位置

      2. 根据历史信息提取出data pattern等信息，设计解决方案

        1. 归纳出四种data pattern

        2. 每一种进行特定优化

          1. direct addressing：数组线性访问

            解决方法：分次数提前预取

          2. indirect addressing：数组随机访问（或者说根据某个变量值访问）

            解决方法：分次数提前预取

          3. unbalanced access：只访问某个结构体的部分成员（原来如此，所以结构体成员需要控制特别精简）

            解决方法：把不常用的数据换到另一个结构体

          4. pointer chasing：一系列结构体chain

            解决方法：把常用数据拉回本结构体

  2. Comments

    1. 分析：
      1. 亮点：自动化根据profiling数据修复程序，开销还低
    2. 启发：
      1. 结合新的硬件特性很重要
      2. 又是一个经典的固定schema+固定fix方法套路
